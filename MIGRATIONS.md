## Overview

**Why Migrations Matter:**

Without version control for database schema:
- ❌ Hard to reproduce database state across environments
- ❌ No rollback capability
- ❌ Team coordination difficult
- ❌ Production deployments risky

**Alembic solves this by:**
- ✅ Tracking schema changes as versioned Python scripts
- ✅ Allowing rollbacks and downgrades
- ✅ Generating migrations automatically from model changes
- ✅ Supporting complex custom migrations

## Setup

### 1. Install Alembic

```bash
pip install alembic
```

Or it's already in `requirements.txt` if you installed dependencies:

```bash
pip install -r requirements.txt
```

### 2. Initialize Alembic (Already Done)

If starting fresh, initialize Alembic:

```bash
alembic init alembic
```

This creates:
```
alembic/
├── versions/          # Migration scripts stored here
├── env.py            # Alembic environment configuration
├── script.py.mako    # Template for new migrations
└── README

alembic.ini           # Alembic configuration file
```

### 3. Configuration Check

**alembic.ini** – Main configuration:
- `sqlalchemy.url`: Database URL (uses `DATABASE_URL` env var at runtime)
- `script_location`: Path to migration scripts (default: `alembic/`)

**alembic/env.py** – Python environment setup:
- Imports app models and metadata
- Reads `DATABASE_URL` from environment
- Configures migration context

## Creating Migrations

### Automatic Migrations (Recommended)

When you **modify `app/models.py`**, Alembic can auto-generate the migration:

```bash
alembic revision --autogenerate -m "Add new_column to employees table"
```

**What gets generated:**
- A new file in `alembic/versions/` with timestamp
- `upgrade()` function: applies the change
- `downgrade()` function: rolls back the change

**Example output:**

```python
def upgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.add_column('employees', sa.Column('new_column', sa.String(), nullable=True))
    # ### end Alembic commands ###


def downgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.drop_column('employees', 'new_column')
    # ### end Alembic commands ###
```

### Manual Migrations

For complex changes (stored procedures, custom logic):

```bash
alembic revision -m "Custom database update"
```

Edit the generated file and add your SQL:

```python
from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    # Custom SQL operations
    op.execute("UPDATE employees SET status = 'active' WHERE status IS NULL")

def downgrade() -> None:
    op.execute("UPDATE employees SET status = NULL WHERE status = 'active'")
```

## Applying Migrations

### Upgrade to Head (Latest Version)

```bash
alembic upgrade head
```

### Upgrade to Specific Version

```bash
alembic upgrade abc123def456
```

### Check Current Migration Status

```bash
alembic current
```

Output: Shows the current migration version

### View Migration History

```bash
alembic history
```

Output:
```
<base> -> 52c1001068a0 (head), Initial migration: create all tables
```

---

## Rollback

### Downgrade One Version

```bash
alembic downgrade -1
```

### Downgrade to Specific Version

```bash
alembic downgrade abc123def456
```

### Downgrade to Base (Remove All Migrations)

```bash
alembic downgrade base
```

⚠️ **Warning:** Downgrade removes tables. Be careful in production!

## Workflow Best Practices

### 1. **Model → Migration → Apply** Workflow

**Step 1:** Modify `app/models.py`

```python
class Employee(Base):
    __tablename__ = "employees"
    id = Column(Integer, primary_key=True)
    email = Column(String, unique=True)  # NEW column
```

**Step 2:** Auto-generate migration

```bash
alembic revision --autogenerate -m "Add unique email constraint"
```

**Step 3:** Review generated file

```bash
# Edit alembic/versions/abc123_xxx.py
# Verify upgrade() and downgrade() look correct
```

**Step 4:** Apply to local database

```bash
alembic upgrade head
```

**Step 5:** Commit and push

```bash
git add alembic/versions/abc123_xxx.py
git commit -m "Migration: Add unique email constraint"
git push
```

### 2. **Team Collaboration**

When pulling new migrations:

```bash
git pull
alembic upgrade head  # Apply all new migrations
```

### 3. **Naming Conventions**

Use clear, descriptive names:

```bash
# ✅ Good
alembic revision --autogenerate -m "Add phone_number column to employees"

# ❌ Avoid
alembic revision --autogenerate -m "Update table"
```

### 4. **Pre-Production Testing**

Test migrations on a **staging database** before production:

```bash
# Staging environment
export DATABASE_URL="postgresql://staging_user:pass@staging-db:5432/hrm_staging"
alembic upgrade head

# Verify data integrity
```

## Troubleshooting

### Error: "Can't import name 'models'"

**Cause:** Python path issue when running Alembic

**Solution:** Run from project root:

```bash
cd /path/to/clinic_hrm_system
alembic revision --autogenerate -m "Migration name"
```

### Error: "no changes detected"

**Cause:** Alembic doesn't see a difference between DB and models

**Solution:**
1. Verify your model change is saved
2. Check `alembic/env.py` imports the right models
3. Manually create migration if needed:
   ```bash
   alembic revision -m "Manual migration"
   ```

### Migration Creates Unwanted Changes

**Cause:** Auto-generation picked up unintended changes

**Solution:**
1. Don't apply the migration yet
2. Edit `alembic/versions/abc123_xxx.py`
3. Remove unwanted `op.add_column()` or `op.drop_column()` lines
4. Apply: `alembic upgrade head`

### SQLite Compatibility Issues

**Cause:** SQLite doesn't support all SQL operations (MODIFY column, DROP constraint, etc.)

**Solution:** Use `render_as_batch=True` in `alembic/env.py` (already configured):

```python
context.configure(
    connection=connection,
    target_metadata=target_metadata,
    render_as_batch=True  # Enables SQLite batch operations
)
```

## CI/CD Integration

### GitHub Actions

Add migration check to your CI pipeline (`.github/workflows/ci.yml`):

```yaml
- name: Check Alembic migrations
  run: |
    alembic current
    alembic history
```

### Cloud Run Deployment

**Option 1: Run migrations during deployment**

```bash
# In your deploy script or Cloud Build step
alembic upgrade head
```

**Option 2: Use Procfile (already configured)**

See `Procfile` in the project root:

```
release: alembic upgrade head
web: uvicorn app.main:app --host 0.0.0.0 --port 8080
```

This runs migrations automatically on each deployment!

## Common Migration Scenarios

### Scenario 1: Add a New Column

```bash
# 1. Update model
# 2. Generate migration
alembic revision --autogenerate -m "Add department_id to employees"

# 3. Apply
alembic upgrade head
```

### Scenario 2: Rename a Column

```bash
# 1. Update model (rename in app/models.py)
# 2. Generate migration
alembic revision --autogenerate -m "Rename employee_code to emp_id"

# 3. Review the generated migration
# 4. Apply
alembic upgrade head
```

### Scenario 3: Add Constraint (Unique, Not Null, etc.)

```bash
# 1. Update model
class Employee(Base):
    email = Column(String, unique=True, nullable=False)

# 2. Generate migration
alembic revision --autogenerate -m "Add unique email and NOT NULL constraints"

# 3. Apply
alembic upgrade head
```

### Scenario 4: Change Column Type

```bash
# 1. Update model
salary = Column(Float)  # Changed from Integer

# 2. Generate migration
alembic revision --autogenerate -m "Change salary column type to Float"

# 3. Apply
alembic upgrade head
```

    # ### commands auto generated by Alembic - please adjust! ###

    pass

    # ### commands auto generated by Alembic - please adjust! ###

## Quick Reference

| Command | Purpose |
|---------|---------|
| `alembic current` | Show current migration version |
| `alembic history` | Show all migrations |
| `alembic heads` | Show latest migration versions |
| `alembic branches` | Show all migration branches |
| `alembic revision -m "msg"` | Create new manual migration |
| `alembic revision --autogenerate -m "msg"` | Auto-generate migration from model changes |
| `alembic upgrade head` | Apply all pending migrations |
| `alembic upgrade <rev>` | Apply migrations up to specific version |
| `alembic downgrade -1` | Rollback one migration |
| `alembic downgrade base` | Rollback all migrations |
| `alembic stamp <rev>` | Mark current DB as version (without running migrations) |
